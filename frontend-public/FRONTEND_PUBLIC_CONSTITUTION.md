# FRONTEND-PUBLIC CONSTITUTION

## 1. Назначение

`frontend-public` — публичная часть персонального сайта Malik Alikberov.

Ее задача:

* представить автора как Java Backend Developer;
* показать проекты, навыки и резюме;
* публиковать статьи и заметки;
* дать посетителю способы связаться с автором;
* быть быстрым, понятным и профессиональным.

Публичный frontend не содержит административных функций, авторизации администратора и управления контентом.

---

## 2. Технологии

Использовать:

```text
React
TypeScript
Vite
React Router
SCSS
Axios
```

Можно добавлять при реальной необходимости:

```text
TanStack Query
React Hook Form
Zod
Lucide React
```

Не добавлять без отдельного решения:

```text
Redux
Zustand
Tailwind CSS
Next.js
UI-kit библиотеки
Microfrontends
```

Перед добавлением новой библиотеки ответить на вопрос:

```text
Какую конкретную проблему текущего frontend-public она решает?
```

---

## 3. Основные маршруты

```text
/                 — главная страница
/about            — обо мне
/projects         — список проектов
/projects/:slug   — страница конкретного проекта
/blog             — список статей
/blog/:slug       — статья
/resume           — резюме
/contacts         — контакты и форма связи
```

Маршруты должны быть понятными, короткими и стабильными.

---

## 4. Структура проекта

```text
src
├── app
│   ├── router
│   └── providers
├── pages
│   ├── home
│   ├── about
│   ├── projects
│   ├── blog
│   ├── resume
│   └── contacts
├── features
│   ├── project-list
│   ├── article-list
│   └── contact-form
├── shared
│   ├── api
│   ├── components
│   ├── hooks
│   ├── types
│   └── utils
└── styles
```

Правила:

1. `pages` собирают страницу из готовых блоков.
2. `features` содержат пользовательские сценарии: список проектов, форма контактов, список статей.
3. `shared` содержит общий переиспользуемый код.
4. Компонент должен иметь одну понятную ответственность.
5. Нельзя создавать компоненты на сотни строк, совмещающие UI, запросы к API и сложную обработку данных.
6. Повторяющиеся элементы выносить в `shared/components`.

---

## 5. Работа с API

1. Frontend получает данные только через Spring Boot REST API.
2. Запросы к API располагаются в `shared/api` или внутри конкретного feature-модуля.
3. UI-компоненты не должны содержать Axios-запросы напрямую.
4. Frontend использует только публичные API endpoints.
5. Не хранить секреты, пароли или административные токены в `frontend-public`.
6. URL backend API передается через environment variables.

Пример:

```text
VITE_API_BASE_URL=http://localhost:8080/api/v1
```

---

## 6. UI и UX

1. Сайт должен корректно работать на телефоне, планшете и desktop.
2. Дизайн: минималистичный, современный, профессиональный.
3. Сначала удобство и читаемость, потом визуальные эффекты.
4. Анимации должны быть редкими, быстрыми и не мешать чтению.
5. У каждой загрузки данных должны быть состояния:

   * loading;
   * error;
   * empty.
6. Формы должны показывать понятные ошибки валидации.
7. Все кнопки и ссылки должны иметь понятные hover, focus и disabled состояния.
8. Текст, кнопки и контраст должны оставаться читаемыми.

---

## 7. Правила контента

1. Главная страница должна за несколько секунд объяснять:

   * кто автор;
   * чем он занимается;
   * какие проекты можно посмотреть;
   * как с ним связаться.
2. Карточка проекта показывает:

   * название;
   * краткое описание;
   * стек;
   * статус;
   * ссылку на GitHub или демо.
3. Страница проекта должна объяснять проблему, решение, стек и ключевые технические решения.
4. Публично отображаются только опубликованные проекты и статьи.
5. Не показывать посетителям черновики, внутренние ID, технические ошибки backend или административные данные.

---

## 8. Стили

1. Использовать SCSS.
2. Общие переменные, reset и базовые стили находятся в `styles`.
3. Стили компонента должны быть максимально локальными и не ломать другие страницы.
4. Не использовать inline styles без конкретной причины.
5. Не копировать одинаковые стили между компонентами — выносить общие решения.
6. Сначала создать базовые дизайн-токены: цвета, отступы, размеры текста, радиусы, тени.

---

## 9. Правила для ИИ-агента

Перед изменением агент обязан:

1. Работать только внутри `frontend-public`, если задача не требует изменений API.
2. Не менять `frontend-admin` и backend без явного требования.
3. Не добавлять библиотеку без объяснения причины.
4. Не внедрять админ-функции, JWT-логику или приватные маршруты.
5. Сохранять существующую структуру `pages / features / shared`.
6. Для новых экранов предусматривать loading, error и empty states.
7. Не ломать существующие маршруты и адаптивность.
8. После изменений проверять, что проект собирается без ошибок.

---

```text
Сначала понятный, быстрый и законченный публичный сайт.
Потом усложнение и новые возможности.
```
